# PeraCRM impersonation phase 3 — write-path audit and hardening

## Executive summary
- Audited the live plugin-owned CRM write paths in `inc/admin/actions.php`, `inc/frontend-data/crm-client-view.php`, `inc/services/activity_service.php`, and the supporting impersonation helpers in `inc/impersonation.php`.
- Hardened reminder, note, activity, client detail quick-action, advisor reassignment, and the obvious deal-owner write paths so actor attribution and advisor ownership are handled separately.
- Standardized the write model to keep **actor** attribution on `peracrm_get_actor_user_id()` while defaulting advisor/assignee ownership to the **effective CRM user** when impersonation is active and no explicit validated owner/advisor was chosen.

## Identity rules now enforced
- **Real user**: the logged-in WordPress user returned by `peracrm_get_real_user_id()`.
- **Actor user**: the audit actor returned by `peracrm_get_actor_user_id()`.
- **Effective CRM user**: the advisor-scoped CRM identity returned by `peracrm_get_effective_crm_user_id()`.
- **Default assignee/owner behavior**:
  - reminder/deal ownership defaults use `peracrm_get_default_assignee_user_id()`
  - explicit advisor/owner submissions go through `peracrm_resolve_assignee_user_id()` or existing advisor validation
  - actor attribution never falls back to the impersonated advisor unless that advisor is also the real actor in a non-impersonated session

## Findings by mutation area

### Reminders
**Previous behavior**
- Reminder creation commonly defaulted `advisor_user_id` to the assigned client advisor or `get_current_user_id()`.
- In impersonated admin sessions, that fallback risked assigning reminder ownership to the real admin rather than the effective advisor.
- Reminder completion/update flows authorized correctly, but activity attribution was not explicitly recorded for status transitions.

**New behavior**
- Reminder create flows now resolve assignee ownership with `peracrm_resolve_assignee_user_id()` so impersonated admin sessions default to the effective CRM user when no explicit advisor is supplied.
- Reminder create and status-change flows now emit activity payloads with `actor_user_id` set from `peracrm_get_actor_user_id()`.
- Pipeline bulk reminder creation now keeps actor/admin attribution separate from advisor ownership and records the reminder owner explicitly in the activity payload.

**Impersonation effect**
- Admin impersonating Mike and adding a reminder from Mike-scoped CRM UI now creates a reminder owned by Mike by default, while reminder activity shows Admin as actor.

### Notes
**Previous behavior**
- Note author storage used the current WordPress user ID directly.
- That happened to keep impersonated admin note authors correct in many flows, but the actor rule was implicit rather than enforced, and note-add activity logging was missing.

**New behavior**
- Note creation now consistently uses `peracrm_get_actor_user_id()` for the stored note author field.
- Note creation now logs `note_added` activity with explicit `actor_user_id` and `note_id`.

**Impersonation effect**
- Admin impersonating Mike can add a note in Mike-scoped CRM, but the note author/activity actor remains Admin rather than Mike.

### Activity logging
**Previous behavior**
- Mutation payloads sometimes set `actor_user_id`, but coverage was inconsistent.
- New event writes could omit actor metadata entirely.

**New behavior**
- `peracrm_log_event()` now backfills `actor_user_id` from `peracrm_get_actor_user_id()` whenever a payload omits it.
- Reminder, note, status-change, advisor-reassignment, and pipeline mutation writes touched in this pass now send explicit actor metadata.

**Impersonation effect**
- Activity rows generated by these write paths remain attributable to the real acting admin during impersonation.

### Client detail mutations / quick actions
**Previous behavior**
- AJAX quick actions for notes/reminders/status updates used `get_current_user_id()` directly and reminder ownership could drift to the real admin fallback.
- Advisor reassign quick action updated ownership but did not explicitly log actor attribution.

**New behavior**
- Client quick actions now separate actor and assignee handling for notes, reminders, reminder status changes, and advisor reassignment.
- Reminder quick actions default to the effective CRM advisor when impersonating.
- Advisor reassign activity now includes the actor user.

**Impersonation effect**
- Mike-scoped quick actions preserve Mike as advisor owner where appropriate, while audit trails show the real admin actor.

### Pipeline / deal mutations
**Previous behavior**
- Pipeline stage-change logging already carried actor metadata in some paths.
- Bulk reminders defaulted to the acting admin when no advisor was chosen.
- Deal owner submissions were accepted without explicit advisor validation in the touched handlers, and convert-to-client defaulted deal ownership to the real current user.

**New behavior**
- Bulk reminder creation now defaults assignee ownership to the effective CRM user when no advisor was chosen.
- Advisor reassignment logs in bulk and client actions now include actor metadata.
- Convert-to-client now defaults deal ownership through the default assignee helper.
- Touched deal create/update handlers now validate explicit `owner_user_id` values against advisor eligibility before writing.

**Impersonation effect**
- Obvious owner/assignee writes in the touched deal/pipeline paths no longer silently collapse actor identity into owner identity during impersonation.

## Remaining limitations
- The deals repository does not currently persist a separate actor/audit column, so this pass hardens owner validation/defaulting in the active handlers rather than introducing a new deal audit schema.
- Reminder storage itself does not have creator/updated-by columns, so the enforced actor attribution currently lives in activity payloads and note author fields rather than in reminder rows.

## Manual QA checklist

### Admin impersonating Mike
- [ ] Create reminder from Mike-scoped view.
  - [ ] Reminder `advisor_user_id` belongs to Mike.
  - [ ] Related activity payload shows Admin as `actor_user_id`.
- [ ] Complete or dismiss reminder from Mike-scoped view.
  - [ ] Reminder actor/log shows Admin.
  - [ ] Reminder advisor ownership does not silently flip to Admin.
- [ ] Add note on a client in Mike-scoped view.
  - [ ] Note attaches to the correct client.
  - [ ] Note author/activity actor shows Admin.
- [ ] Run pipeline bulk “add reminder” while viewing Mike-scoped CRM.
  - [ ] Reminder assignee defaults to Mike unless an explicit valid advisor is chosen.
  - [ ] Activity actor remains Admin.

### Non-impersonated admin
- [ ] Add reminder from admin CRM view and confirm existing behavior still works.
- [ ] Add note and confirm note author/activity actor stays the current admin.
- [ ] Reassign advisor and confirm activity still records the change.

### Employee
- [ ] Employee adds reminder in normal mode and keeps existing ownership behavior.
- [ ] Employee adds note in normal mode and remains the note author/activity actor.
- [ ] Employee reminder status updates still authorize and save normally.
